## Color

Full token set inherited from Phase 9 (`globals.css`). No new tokens introduced.

| Role | Token | OKLCH Value | Usage |
|------|-------|-------------|-------|
| Dominant (60%) | `--background` | `oklch(0.99 0 0)` | Page background, table row background |
| Secondary (30%) | `--muted` | `oklch(0.97 0 0)` | Sidebar, table header row |
| Accent (10%) | `--fluent-accent` | `oklch(0.52 0.22 261)` | Active sidebar nav item only — NOT used on Exclude button |
| Destructive | `--destructive` | `oklch(0.577 0.245 27.325)` | Not used in this phase — exclusion is non-destructive |

**Accent reserved for:** Active/selected sidebar nav item only. The Exclude action is NOT styled as accent — it is `variant="ghost"`, visually neutral.

**Why not destructive?** Excluding an object is reversible from the Exclusions page (EXCL-03). Destructive red styling would misrepresent the action's risk level.

**Already-excluded row dimming:** `opacity-60` — matches the existing `isExcluded` pattern already implemented in `ReclassifyPage.tsx` and consistent with the pattern expected in `ObjectTable`.

**Tier badge colors** (pre-existing, unchanged):
- ControlPlane: `text-tier-control border-tier-control`
- ManagementPlane: `text-tier-management border-tier-management`
- UserAccess: `text-tier-user border-tier-user`
- Unclassified: `text-tier-unclassified border-tier-unclassified`

## Screen Layout: ObjectTable (ObjectBrowser)

### Current columns (6): Display Name | Object Type | Tier | RBAC System | PIM Type | On-Prem Sync
### Phase 10 adds: Actions column (7th, rightmost)

```
┌──────────────────────────────────────────────────────────────────────┐
│ | Display Name | Object Type | Tier | RBAC | PIM | Sync | Actions  │
├──────────────────────────────────────────────────────────────────────┤
│ | John Smith   | User        | [CP] | AzAD | Elig| Cloud| [Exclude]│
│ | SVC-Acct-01  | SvcP        | [MP] | AzAD | Perm| —    |[Excluded]│  ← already excluded
└──────────────────────────────────────────────────────────────────────┘
```

**Actions column header:** `"Actions"` — `TableHead` with `text-right` alignment, `w-[100px]` fixed width.

**Non-excluded row:** Ghost button — `<Button variant="ghost" size="sm" className="h-7 gap-1">`
  ```tsx
  <ShieldMinus size={14} />
  Exclude
  ```
  `onClick` must `event.stopPropagation()` to prevent row click from opening the detail panel.

**Already-excluded row:**
  - Entire row: `className="opacity-60"` (dim, consistent with ReclassifyPage pattern)
  - Actions cell: `<Badge variant="outline" className="text-xs text-muted-foreground border-muted-foreground/40">Excluded</Badge>`
  - No button — the action is not repeatable; removing exclusion is done on the Exclusions page

**Loading state:** `onExclude` prop is optional; during the POST request the button should be `disabled` with `Loader2` icon replacing `ShieldMinus`. Button returns to normal after the toast resolves (optimistic disable — re-enable on error).

**ObjectTableProps extension:**
```ts
onExclude?: (object: PrivilegedObject) => void;
excludedIds?: Set<string>;  // or integrate with useExclusions at ObjectBrowser level
```

## Interaction States

| State | Visual | Behavior |
|-------|--------|----------|
| Default (not excluded) | Ghost button with `ShieldMinus` + "Exclude" | Click triggers POST, button disabled during request |
| In-flight (POST pending) | Button disabled, `Loader2` replaces icon | Prevents double-submit |
| Success | Toast: "Object excluded" + "View Exclusions →" action link | Row becomes `opacity-60`, button replaced by "Excluded" badge |
| Error | Toast: "Failed to exclude object" | Button re-enabled, row state unchanged |
| Already excluded | `opacity-60` row, "Excluded" badge (ObjectTable) or badge in Object cell (ReclassifyPage) | No button rendered |

**Double-exclude prevention:** After a successful exclude, the button is immediately replaced by the "Excluded" badge. This is achieved by the `useExclusions` hook's `invalidate()` call after a successful POST, which refreshes the `exclusions` Set used by both pages.

## API Contract

Phase 10 requires one new server endpoint (all others from Phase 9 already exist):

| Method | Path | Body | Response | Usage |
|--------|------|------|----------|-------|
| POST | `/api/exclusions` | `{ "guid": "<uuid>" }` | 201 Created or 409 Conflict (already excluded) | Add exclusion from ObjectBrowser or ReclassifyPage |

**POST behavior:**
- Validate UUID with Zod
- If already present: return 409 (client should handle gracefully — treat as success for UX purposes, show toast "Already excluded")
- If absent: append to `ExcludedPrincipalId[]`, atomicWrite to Global.json, return 201
