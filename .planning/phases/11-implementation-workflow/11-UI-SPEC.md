## Page Architecture Decision

**Single page with state machine** вҖ” preferred (matches `RunCommandsPage` pattern).  
No separate routes per step. No dialog overlays for confirmation.

| State | Screen | Trigger |
|-------|--------|---------|
| `idle` | Action Selection | Page load / "Apply Again" |
| `confirming` | Pre-run Confirmation (inline) | "Review & Apply" clicked |
| `running` | Live SSE Log | "Run Now" clicked |
| `done` | Outcome Summary | All cmdlets exit OR stop |

Route: `/apply`  
Component: `ApplyPage.tsx`  
Layout wrapper: `p-6 max-w-2xl mx-auto` (focused task flow; narrower than Browse Objects `max-w-7xl`)

## Implementation Notes for Planner

1. **Sequential SSE execution:** `ApplyPage` runs cmdlets one by one. For each selected cmdlet, call `POST /api/commands/run` with that cmdlet's name and no parameters (or TenantName/SubscriptionId from config). Wait for the `exit` SSE event, record exit code, then start the next cmdlet.

2. **Per-cmdlet result tracking:** Maintain a `results: Record<AllowlistedCmdlet, 'pass' | 'fail' | 'pending' | 'skipped'>` map. Populate as each cmdlet exits.

3. **Terminal separator pattern:** Reuse the existing gray separator injected in `RunCommandsPage`:  
   `"в”Җв”Җв”Җ {FriendlyName} В· {timestamp} в”Җв”Җв”Җ"` in `text-zinc-500`.

4. **Default selection:** All 4 actions checked on mount (apply everything by default). User can uncheck to exclude.

5. **Config params:** Fetch `/api/config` on mount to pre-populate TenantName/SubscriptionId for the confirmation table. These are passed as cmdlet parameters just as RunCommandsPage does.

6. **"Apply Again"**: Reset to `idle` state preserving the last-used selection (not clearing to default). Clears `htmlContent` and `results`.

7. **Phase 12 hook:** The state machine's `confirming` screen will gain a "Dry Run" toggle in Phase 12 (IMPL-05 / `SampleMode`). Reserve space in the confirmation screen layout for this addition вҖ” it will slot in as a checkbox row above the warning Alert.
