<remaining_work>
**Current blocker (step 4 — Classifying):**

Error: `Authentication needed. Please call Connect-MgGraph`

The `Save-EntraOpsPrivilegedEAMJson` cmdlet calls Graph APIs, but `Connect-MgGraph` auth tokens live in the PowerShell session that ran `Connect-EntraOps` — which is a completely separate process from the classify spawn. The session/token is not persisted to disk by default.

**Root cause to investigate:**
- Does `Connect-MgGraph` support `-UseDeviceAuthentication` or token caching that survives across processes?
- Does `Connect-EntraOps` offer an `-AlreadyAuthenticated` or token-passing mode that could be used — e.g. get token from the connect process and pass `MsGraphAccessToken` + `AzArmAccessToken` params to a second process?
- Look at `Connect-EntraOps.ps1` params: `$AzArmAccessToken`, `$MsGraphAccessToken` — these exist and could be retrieved via `Get-AzAccessToken` + `(Get-MgContext).AuthToken` at end of the connect stream, stored server-side, then passed to the classify invocation.
- Alternative: run both connect AND classify in the same persistent pwsh process (keep the connect process alive, pipe classify commands into it). This is architecturally larger but solves the root cause permanently.

<blockers>
**Active blocker:** MgGraph auth session doesn't survive to the classify pwsh process.

The connect process authenticates (Connect-AzAccount + Connect-MgGraph) but those sessions are in-process only. The classify process spawns fresh and has no auth context.

**Approaches to explore (in order of preference):**
1. After auth stream ends, server calls `Get-AzAccessToken -ResourceTypeName MSGraph` + `Get-AzContext` in a *third* pwsh to retrieve tokens, stores them in memory, then passes `$AzArmAccessToken` and `$MsGraphAccessToken` to `Connect-EntraOps -AuthenticationType AlreadyAuthenticated` at the start of the classify command.
2. Keep the connect pwsh process alive (don't kill it on exit) and pipe classify commands into it — reuses the session. Requires architectural change to connect service.
3. Use `Connect-MgGraph -UseDeviceAuthentication` again at start of classify process — prompts user a second time (bad UX, but works as a fallback).
</blockers>

<context>
Phase 4 is the "Connect & Classify" wizard. The fundamental challenge is that this module was designed to run interactively in a single persistent PS session — the GUI is trying to drive it as discrete process invocations which breaks session-scoped auth state.

The cleanest solution is probably approach 1: after connect stream finishes (exit code 0), spawn a quick pwsh to extract the Az access token and MgContext token, hold them in the server's connect service state, then prepend `Connect-EntraOps -AuthenticationType AlreadyAuthenticated -AzArmAccessToken ... -MsGraphAccessToken ...` to the classify command.

This requires:
- Updating `connect.ts` `runConnect` to extract tokens after successful connect
- Storing tokens in connection state alongside `tenantName`
- Passing tokens to `buildPwshArgs` / the classify invocation
- The `ConnectPage.tsx` classify call doesn't need to change (server handles it)
</context>
